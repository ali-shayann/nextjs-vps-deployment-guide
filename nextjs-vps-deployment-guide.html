How to Deploy a Next.js App on a VPS with Apache

A no-fluff, beginner-friendly guide using PM2, cPanel, and a proper reverse proxy

5 min read · DevOps · Next.js · Apache

You've built a Next.js app. Now what? Getting it live on a real server trips up a lot of developers. This guide walks you through the whole thing — from SSH to a working HTTPS domain — in under 30 minutes.

// deployment pipeline
💻 Your Local Machine
git push
🐙 GitHub Repository
SSH in + git clone
⚡ VPS Server
📦 Install Node.js via NVM
🔧 npm install + npm run build
🔄 PM2 — App running on :3000
reverse proxy
🌐 Apache / cPanel WHM
📁 Create userdata/proxy.conf
🔁 Copy config for SSL vhost
⚙️ Rebuild Apache config
🔒 SSL via AutoSSL / Let's Encrypt
🌍 yourdomain.com — Live on HTTPS

What you need before starting

You'll need a Linux VPS (AlmaLinux, Ubuntu, or Debian all work), a domain pointed at your server's IP, cPanel/WHM access, and your Next.js app pushed to a GitHub repo. That's it.

Step 1 — SSH into your server

Open Terminal and connect:

ssh your_username@your_server_ip

If you're on cPanel, there's also a browser terminal at https://YOUR_IP:2083 → Advanced → Terminal.

Step 2 — Install Node.js with NVM

NVM is the cleanest way to manage Node on a server — no sudo headaches.

curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.39.7/install.sh | bash
source ~/.bashrc
nvm install --lts
nvm use --lts

Step 3 — Clone, build, and start with PM2

git clone https://github.com/YOU/YOUR_REPO.git
cd YOUR_REPO
nano .env.local          # add your env vars
npm install
npm run build
npm install -g pm2
pm2 start npm --name "my-app" -- start
pm2 save && pm2 startup
Important After pm2 startup, copy and run the full sudo env PATH=... command it prints. Without it, your app won't survive a reboot.

Test it's working before touching Apache:

curl http://localhost:3000

Step 4 — Configure Apache as a reverse proxy

This is where most tutorials go wrong. On cPanel/WHM servers, you should never edit vhost files directly — they get overwritten on every rebuild. Instead, use the userdata directory:

1

Create the config directories

Replace cpanelusername and yourdomain.com with yours.

mkdir -p /etc/apache2/conf.d/userdata/std/2_4/cpanelusername/yourdomain.com
mkdir -p /etc/apache2/conf.d/userdata/ssl/2_4/cpanelusername/yourdomain.com
2

Create the proxy config

nano /etc/apache2/conf.d/userdata/std/2_4/cpanelusername/yourdomain.com/proxy.conf

Paste this inside:

<IfModule mod_proxy.c>
    ProxyPreserveHost On
    ProxyPass / http://127.0.0.1:3000/
    ProxyPassReverse / http://127.0.0.1:3000/
</IfModule>
3

Copy the config for SSL

cp /etc/apache2/conf.d/userdata/std/2_4/cpanelusername/yourdomain.com/proxy.conf \
   /etc/apache2/conf.d/userdata/ssl/2_4/cpanelusername/yourdomain.com/
4

Rebuild Apache

/scripts/ensure_vhost_includes --all-users
/scripts/rebuildhttpdconf
/scripts/restartsrv_httpd
Why this approach? Files in userdata/ survive cPanel/WHM updates and Apache rebuilds. Editing vhosts directly gets overwritten. This is the correct, production-safe way.

Step 5 — Enable SSL

In cPanel, go to Security → SSL/TLS Status and click Run AutoSSL. A free Let's Encrypt certificate will be applied within a minute or two. Done.

Deploying updates

Every time you push new code, SSH in and run:

cd ~/YOUR_REPO && git pull && npm install && npm run build && pm2 restart my-app

Or save the above as a deploy.sh script and run it with a single ./deploy.sh.

Quick troubleshooting

If something isn't working, these commands will tell you almost everything you need to know:

pm2 logs my-app                              # app errors
curl http://localhost:3000                   # is the app responding?
tail -f /usr/local/apache/logs/error_log    # apache errors
lsof -i :3000                               # is port 3000 in use?

The most common issue is the proxy config not applying after an Apache rebuild. Just re-run /scripts/ensure_vhost_includes --all-users and rebuild again.
